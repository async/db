Contracts grow from evidence.

Async DB infers practical local contracts from records first. Schema files become the upgrade path for required fields, descriptions, defaults, uniqueness, relations, and stricter diagnostics.

Resource contract: users

collection id: id
Field Type Contract use
id string Stable document id, REST lookup key, generated TypeScript property.
email string Can become unique and required when a schema source is added.
role enum Inferred from fixtures or declared explicitly with defaults.
authorId relation Can feed relation metadata for REST expansion and viewer context.

Schema source when inference is not enough

explicit contract 
{
  "kind": "collection",
  "idField": "id",
  "fields": {
    "email": {
      "type": "string",
      "required": true,
      "unique": true,
      "description": "Email address used for sign-in."
    },
    "role": {
      "type": "enum",
      "values": ["admin", "user"],
      "default": "user"
    }
  }
}

types + routes 
// REST
await fetch("/db/users.json");

  method: "POST",
  body: JSON.stringify({

    email: "grace@example.com"
  })
});

// GraphQL
query {
  users {
    id
    name
    role
  }
}

// generated type
type User = {
  id: string;
  name: string;
  email: string;
  role?: "admin" | "user";
};

Production is a resource decision.

The app does not need a data-layer rewrite when one resource gets serious. Keep the contract stable, move the store behind it, and put production-facing traffic behind registered operations and app-owned controls.

Resource choice Use when Keep stable Production boundary
core

JSON file store

Settings, flags, content, templates, plan definitions, policy rules, and low-write project data.

Readable source fixtures, generated state under .db/state, and optional sourceFile writeback.

Single-writer file-suitable resources that can live with project-local state.

structured

SQLite or Postgres

High-write, multi-writer, transactional, or compliance-heavy resources.

Async DB metadata, resource names, generated types, and route-facing contracts.

App-owned database clients and app-owned auth, limits, and observability.

distributed

KV and Redis-like stores

Caches, lookups, and app-owned infrastructure where JSON file state is not the right fit.

Contracts visible to docs, tools, viewer metadata, and the same resource model.

Store behavior chosen per resource, without turning Async DB into a generic driver layer.

hardened

Registered operations

Production traffic that should not expose raw writable dev routes.

Allowlisted operation refs and reviewed request/response shapes.

App policy, session context, authorization, rate limits, and logs.
